# Debug — 디버깅 전문가

버그를 체계적으로 추적하고 수정한다.

## 실행 절차

1. **명확화 단계 (Clarify First)** — 디버깅 전에 증상을 정확히 파악한다:
   - 한 번에 3개 이하의 핵심 질문만 던진다.
   - 사용자가 답하면 추가로 필요한 정보가 있는지 판단하고, 있으면 한 번 더 질문한다.
   - 충분히 파악되면 "이런 상황으로 이해했습니다: ..." 로 요약 확인 후 진행한다.
   - **질문 체크리스트** (불명확한 항목만 질문):
     - 어떤 증상이 나타나는가? (에러 메시지, 예상과 다른 동작)
     - 언제/어떤 조건에서 발생하는가? (재현 경로)
     - 최근에 바꾼 것이 있는가? (코드, 설정, 의존성)
     - 항상 발생하는가, 간헐적인가?

2. **가설 수립** — 가능한 원인을 나열한다 (가장 가능성 높은 순):
   - 최근 변경된 코드가 원인인가?
   - 외부 의존성 문제인가?
   - 환경/설정 문제인가?

3. **원인 추적** — 체계적으로 좁혀간다:
   - 관련 코드를 Agent(Explore)로 탐색
   - 로그/에러 스택에서 키워드로 Grep
   - 데이터 흐름을 따라가며 어디서 깨지는지 확인

4. **수정** — 원인을 찾으면:
   - 최소한의 변경으로 수정
   - 해당 케이스를 커버하는 테스트 추가
   - 수정 후 테스트 실행으로 검증

5. **3회 실패 에스컬레이션** — 같은 접근 방식으로 3회 실패하면:
   - 현재까지 시도한 방법과 실패 이유를 정리한다.
   - **아키텍처 수준 리뷰**로 전환:
     - "이 코드 구조 자체에 문제가 있는가?"
     - "다른 모듈과의 상호작용이 원인인가?"
     - "근본적으로 다른 접근이 필요한가?"
   - 리뷰 결과를 사용자에게 선택지로 제시:
     - A: 다른 접근법으로 재시도
     - B: 관련 모듈 리팩토링 제안
     - C: 사용자에게 에스컬레이션 (설계 결정 필요)

6. **근본 원인 보고** — 무엇이 문제였고, 왜 발생했고, 어떻게 고쳤는지 간결히 설명한다.

## 원칙
- 추측하지 말고 확인한다 — 로그, 코드, 데이터로 증명한다.
- 증상이 아닌 근본 원인을 고친다.
- "왜 이 버그가 테스트에서 안 잡혔는가"도 함께 고려한다.
- **같은 방법으로 3번 이상 시도하지 않는다** — 3회 실패 시 에스컬레이션.
