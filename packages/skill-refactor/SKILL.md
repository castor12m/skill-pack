# Refactor — 리팩토링 전문가

기존 코드의 구조를 개선하되 동작은 변경하지 않는다.

## 실행 절차

1. **명확화 단계 (Clarify First)** — 리팩토링 전에 모호한 점을 질문으로 좁혀간다:
   - 한 번에 3개 이하의 핵심 질문만 던진다.
   - 충분히 명확해지면 "이렇게 리팩토링하겠습니다: ..." 로 요약 확인 후 진행한다.
   - **질문 체크리스트** (불명확한 항목만 질문):
     - 리팩토링 대상은? (특정 파일/함수/모듈)
     - 어떤 문제를 개선하고 싶은가? (중복, 복잡도, 네이밍, 구조)
     - 리팩토링 범위의 제한이 있는가? (이 파일만 / 관련 파일 전체 허용)
     - 기존 테스트가 있는가? (동작 보전 검증 방법)

2. **대상 파악** — 리팩토링할 코드를 읽고 현재 문제점을 분석한다:
   - 중복 코드
   - 과도한 복잡도 (긴 함수, 깊은 중첩)
   - 불명확한 네이밍
   - 잘못된 추상화 수준

2. **영향 범위 확인** — 해당 코드를 참조하는 모든 곳을 Grep/Agent로 찾는다:
   - import/require하는 파일
   - 테스트 파일
   - 타입 정의 참조

3. **리팩토링 계획** — 변경 전에 계획을 제시한다:
   - 무엇을 왜 변경하는지
   - 동작 변경이 없음을 어떻게 보장하는지

4. **리팩토링 실행** — 단계적으로 수행한다:
   - 한 번에 하나의 리팩토링만 적용
   - 각 단계 후 테스트 실행으로 동작 보전 확인

5. **검증** — 기존 테스트가 모두 통과하는지 확인한다.

## 원칙
- 동작 변경 금지: 리팩토링과 기능 변경을 절대 섞지 않는다.
- 테스트가 없는 코드는 테스트를 먼저 추가한 후 리팩토링한다.
- 과도한 추상화를 피한다 — 3번 반복되기 전까지는 중복을 허용한다.
